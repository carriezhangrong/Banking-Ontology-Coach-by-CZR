# 知识工程术语表（讲解前先读，再翻译成学员能懂的话）

每个词条给出：严谨定义 + 大白话 + 判据/银行例子。讲给新手时**先用大白话和例子，必要时再引入正式名称**。

## 类 / 概念（Class）
一类共享相同特征的事物的集合。
- 大白话：一种"类别""种类"。
- 银行例：`法人实体`、`账户`、`金融工具`、`协议` 都是类。

## 实例 / 个体（Instance / Individual）
某个类的具体成员。
- 判据（类 vs 实例）：能不能再往下分出"不同的它"？能 → 多半是类；指独一无二那一个 → 实例。
- 银行例：`法人实体` 是类；"某银行（LEI: 5493001...）"是它的实例。

## 类层级 / 上下位关系（is-a / subsumption）
类之间"是一种"的关系，子类继承父类全部属性。
- 判据口诀："凡是 X 都是一种 Y"成立，X 才是 Y 的子类。
- 银行例：`储蓄账户 is-a 账户`、`法人实体 is-a 当事方`。
- 反例：`分行 is-a 银行` ❌——分行是银行的一部分（part-of），不是一种银行。

## 属性（Property）——分两种
### 数据属性（Datatype property）
把实例连到一个**字面值**（数字/文本/日期/布尔），描述特征。
- 银行例：`成立日期 → 日期`、`账户余额 → 金额`、`LEI 代码 → 文本`。
### 对象属性（Object property）
把实例连到**另一个实例**，表达概念间关系。
- 银行例：`持有（客户 → 账户）`、`隶属于（分行 → 法人实体）`。
- 技巧：箭头那头是"一个值"还是"另一个概念"？值→数据属性；概念→对象属性。

## 属性的刻面（Facets）
- **定义域 domain**：属性属于哪个类。
- **值域 range**：属性指向哪个类/哪种值。
- **基数 cardinality**：能取几个值（如"一个账户恰好隶属 1 个法人实体"）。

## 对象属性的特性（Characteristics）
- **函数性 functional**：每实例最多一个值（如"开户法人"）。
- **对称 symmetric**：A 关系 B 则 B 关系 A（如"关联方"）。
- **传递 transitive**：A→B 且 B→C 则 A→C（如受益所有权的"间接控制"链）。
- **反函数/逆 inverse**：成对反向关系（如"持有"与"被持有于"）。

## 关系的三种典型类型（务必让学员分清）
- **is-a（上下位）**：子类→父类，"是一种"。
- **part-of（整体-部分）**：组成关系，"是…的一部分"（如分行 part-of 法人实体）。
- **普通关联（association）**：其余领域关系，如"持有""担保""作为对手方参与"。

## 角色 vs 类型（银行建模的关键范式，详见 fibo-reference）
"对手方""借款人""受益所有人"不是机构/人的内在类别，而是某**当事方（Party）在某合约/情境中扮演的角色**。FIBO 用 PartyInRole 模式表达，别把它们立成独立类型。

## 公理 / 约束（Axiom / Constraint）
领域里**必须成立**的逻辑规则，让本体可被机器一致性检查——这是它区别于普通词汇表、能支撑监管报送的核心。
- **不相交 disjointness**：两类无共同实例（如 `自然人` 与 `法人实体` 不相交）。
- **基数约束**：如"每笔交易必须恰好有 1 个发起方"。
- **定义域/值域限制**：限定属性只能连接特定类。

## 胜任力问题（Competency Questions, CQ）
一组"建好的本体必须能回答"的自然语言问题，用来界定范畴、最后验收；银行场景常追溯到监管/治理需求。
- 银行例："某对手方的最终受益所有人是谁？""某法人实体持有哪些账户？""哪些交易需纳入某监管报送口径？"
- 用法：先写 CQ 定边界，建模对照取舍，收尾逐条检验本体能否答出。
